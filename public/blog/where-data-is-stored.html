Where Your Data Actually Lives: A Primer on Volatile and Persistent Memory

"Did you wipe the RAM?" is one of the most common questions we hear from security teams. It is also one of the easiest to answer �?but the question that should be asked instead is much more interesting.

The question we keep hearing

"Can you certify that the RAM in our laptops has been wiped?"

The short answer is yes �?and so can you. Unplug the laptop and walk away for a minute. RAM is wiped. Done.

The longer answer is that the question itself reveals a common misunderstanding about how modern computers store information. RAM is not the place security teams need to worry about. The places they should worry about are the ones nobody asks us about. So let us walk through where data actually lives inside the equipment you are retiring, and what it takes to make sure the sensitive pieces are properly removed.

Volatile memory: gone at power-off

Volatile memory is any storage that requires continuous power to retain its contents. Cut the power, the bits decay within seconds to minutes, and whatever was there is gone. The major examples inside the hardware you handle every day:

  • System RAM (DDR4, DDR5) �?the main working memory used by the operating system and applications.
  • GPU memory (GDDR, HBM) �?the dedicated framebuffer and compute memory on a discrete graphics card.
  • CPU caches (L1, L2, L3) �?small high-speed buffers built into the processor.
  • Network device buffers �?short-lived packet queues inside switches, NICs, and routers.

All of this is wiped by the simple act of disconnecting power. There is no sanitization step required, no software tool to run, no certificate to issue. A laptop that has been unplugged with its battery removed for thirty seconds has volatile memory that contains nothing recoverable.

Exception worth knowing A handful of academic attacks (cold-boot attacks) have demonstrated that DRAM cells can retain a partial state for tens of seconds at room temperature, and longer if chilled. These attacks are real but extremely narrow in scope, require physical access while the machine is still warm, and are not a concern for normal end-of-life processing.

Persistent storage: what survives

Persistent storage is the opposite: the bits remain whether the power is on or off. This is where the actual data sanitization work happens. The categories you will encounter on enterprise hardware:

  • Hard disk drives (HDD) �?magnetic platters. Older but still common in storage arrays.
  • Solid-state drives (SATA, SAS SSDs) �?flash memory packaged in a 2.5" form factor.
  • NVMe drives (M.2, U.2) �?flash memory connected directly to the PCIe bus.
  • eMMC and UFS storage �?flash soldered onto the system board in thin laptops, tablets, and smartphones.
  • Tape media �?LTO cartridges used for archival backups.

These are the surfaces that require deliberate sanitization. A drive that has been powered off retains every byte of its contents until those bytes are overwritten, cryptographically erased, or the medium is physically destroyed. Pulling the power does nothing to protect the data on a hard drive.

The places people forget

This is the part of the conversation that usually surprises people. Persistent storage is not limited to the obvious drives. Modern hardware contains many small persistent regions that are easy to overlook:

BIOS / UEFI firmware

Stored in a small flash chip on the motherboard. Holds boot settings and sometimes configuration data �?including, in some enterprise deployments, embedded credentials for management interfaces.

Baseboard management controllers

Servers contain a separate management processor (iDRAC, iLO, IPMI) with its own flash storage holding network configuration, user accounts, and SSL certificates. Wiping the host operating system does nothing to this.

Network device configurations

Switches and routers retain their running configuration �?including ACLs, VLAN assignments, SNMP community strings, and sometimes credentials �?in onboard flash. Powering them down preserves all of it.

SSD over-provisioning regions

SSDs reserve a portion of their flash for wear-leveling and bad-block replacement. Data written to the drive may end up in cells that are no longer addressable by the operating system. A simple block-level overwrite cannot guarantee these regions are touched, which is why SSDs require either the drive's own sanitize command or cryptographic erase.

HPA and DCO regions on HDDs

Older hard drives can have hidden areas �?Host Protected Area and Device Configuration Overlay �?that are not visible to the operating system but can contain data. Proper sanitization tools detect and address these.

Multifunction printer hard drives

Easily forgotten. Enterprise copiers and printers often contain a hard drive that caches every scanned, printed, and faxed document. They need to be sanitized before disposal just like a server.

Why this matters for sanitization

If your mental model of "wipe the data" stops at the visible C: drive, you have a gap. A device that has had its primary storage drive wiped may still contain recoverable information in any of the surfaces above. Whether this matters depends on what kind of data your organization handles �?but for regulated environments (healthcare, finance, defense, government), it matters a great deal.

The standard most enterprises use to address this is NIST SP 800-88r1, which defines the recognized techniques and categories of media sanitization. A serious sanitization program does not just run a wipe utility on the boot drive and call it done. It inventories every persistent surface inside the device and applies an appropriate method to each.

What a serious vendor covers

When you evaluate an ITAD partner, the questions worth asking are not about RAM. They are:

  1. Does the workflow identify every persistent storage surface on each device class you send (servers, laptops, switches, MFPs)?
  2. Are SSDs sanitized using the drive's native sanitize command or cryptographic erase, not block-level overwrite alone?
  3. Are baseboard management controllers and BIOS settings reset on servers?
  4. Are network device configurations cleared before resale or recycling?
  5. Do you receive a per-device certificate that names the specific storage media sanitized and the method used?

The answer to "did you wipe my RAM?" should always be yes, accompanied by a small, polite explanation of why the more important questions are the ones above.
